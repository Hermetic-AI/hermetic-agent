OpenCode + MCP + Skill + AGUI

基于 OpenCode 的差旅 Agent 技术演示

用开源 Agent Runtime 承接通用智能体能力，用协议化工程把国内机票查询、预订、政策与动态 UI 落到可维护系统中。

不自研 Agent Runtime
不把差旅流程硬编码进核心
不让 UI 依赖自然语言猜测
OpenAgent 差旅 Agent 技术方案1 / 10
01 技术定位

这不是 Prompt Demo，而是可落地的差旅 Agent 工程范式

核心价值在于把 Agent 的通用能力、差旅业务事实、流程约束和前端交互拆开，让每一层都能独立演进。

OpenCode 承接执行底座与模型生态
MCP 接入真实差旅业务能力
Skill 固化流程、状态与工具边界
AGUI 输出结构化动态 UI
定位2 / 10
02 总体架构

轻量分层：Agent Core 不进业务分支

差旅能力通过 Skill、MCP 和卡片协议注入，OpenAgent Hub 负责场景路由、事件映射和动态 UI 转换。

总体架构3 / 10
03 开发范式

三层协议约束，把 Agent 行为收敛到业务边界内

MCP 决定能调用什么，Skill 决定什么时候调用，AGUI 决定如何呈现和继续交互。

协议化带来的工程结果

边界清晰业务事实不由模型编造，最终以 MCP / Java runtime 为准。
行为可审计每个阶段允许的工具、必填参数、异常恢复都有显式约束。
体验稳定前端按 card_type 渲染，避免自然语言输出驱动关键 UI。
扩展轻场景能力插拔，不需要改 OpenCode Core。
协议范式4 / 10
04 端到端链路

首轮查询速度优先：三要素齐全，直接查航班

用户给出出发地、目的地、日期后，Skill 约束 Agent 不先追问舱位，不绕路，直接调用航班查询工具并生成结构化卡片。

快速查询链路5 / 10
05 预订状态机

用状态机约束流程，用 Java Runtime 保留权威副作用

Skill 只做高层阶段约束和防错；真正的状态、政策、价格、订单预览仍由业务系统负责。

状态机6 / 10
06 动态 UI 协议

从对话到可操作界面：关键结果都走 Card Schema

前端不解析大段自然语言，不猜业务含义，而是按白名单 card_type 渲染对应组件。

OD_INPUT缺少出发地、目的地或日期时补充信息
FLIGHT_RESULT航班结果摘要与推荐方案
CABIN_LIST舱位与价格选择
PASSENGER_FORM乘机人信息补全
OAT_BINDING出差单、成本中心、联系人绑定
PRICE_VERIFY价格变化确认
POLICY_DECISION政策超规决策
ORDER_CONFIRM订单预览与最终确认
动态 UI7 / 10
07 方案优势

优势不是“能跑”，而是复杂业务可以长期维护

差旅流程天然包含政策、价格、订单、乘机人、成本中心和异常恢复。该方案把复杂度放在合适的位置。

高内聚 国内机票预订约束集中在 Skill、Scenario 与 Card Schema 中。
低耦合 OpenCode、Hub、Skill、MCP、前端各司其职，边界清楚。
可维护 状态机、工具契约、工作流和 UI 协议都以文件化方式沉淀。
易扩展 新增场景优先新增 Skill 与协议，不改 Agent Core。
方案优势8 / 10
08 我的优势

我真正交付的是一套可复用的 Agent 业务落地方法

重点不在“接了一个模型”，而在于把模型能力、业务系统、流程治理和前端体验组织成完整工程闭环。

架构判断力

取舍复用 OpenCode，不重复造 Runtime。
边界Skill 不做影子订票引擎，Java 保持业务权威。
治理用协议约束 Agent，而不是只靠提示词。

工程落地力

闭环从用户输入到 MCP，再到 SSE 卡片输出。
兜底Hub 可从工具结果自动组装航班卡片。
测试覆盖卡片转换、事件映射和 token 配置。

业务扩展力

场景机票查询、预订、政策恢复可逐步扩展。
协议新业务通过新 Skill、Tool 和 Card 接入。
体验卡片化交互让 Agent 更像产品，而不是问答框。
我的优势9 / 10
