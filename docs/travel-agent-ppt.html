05 · 方案优势

四大优势：高内聚 · 低耦合 · 易维护 · 易扩展

不是 Prompt 调优，而是一套协议化架构。把复杂差旅流程拆成可审计、可测试、可插拔的模块。

高内聚

差旅约束集中沉淀到 Skill 目录、scenario yaml、ask_user schema。流程顺序 · Tool 调用时机 · 参数约束 · 状态机 · 卡片结构 · 工作流说明，一处定义，多处生效。

低耦合

OpenCode Runtime · Hub · Skill · MCP · Java · 前端，层与层之间只通过协议字段交互。业务侧不侵入 OpenCode 核心，UI 不依赖自然语言输出。

强可维护

所有关键约束以文件形式存在：路由 yaml · 状态机 json · 契约 json · 卡片 schema · 工作流 md。改一处即可生效，团队协作零负担。

低扩展成本

新差旅流程 = 新 Skill + 新 Tool Contract + 新 Card Schema + 新 scenario yaml。Agent Core 与现有 Skill 不动，实现即插即用。

06 · 核心 Skill 深潜

薄 Skill：约束流程，不做影子订票引擎

Skill 只做四件事：Sequence · Normalize · Compact · Guide。真实业务状态与副作用仍由 Java booking runtime 权威执行。

Skill 职责 · Sequence / Normalize / Compact / Guide

编排调用顺序 · 归一化日期与枚举 · 压缩 MCP 大结果 · 通过 schema 与 workflow 约束 Agent 行为。

不做的事 · 权威层负责

不查航班 · 不算票价 · 不判政策 · 不写 Redis 状态 · 不保存订单预览。所有副作用由 Java booking runtime 兜底。

07 · 端到端时序

从一句话到航班卡片：四跳协议转化

「北京到上海明天的机票」→ 提取 · 归一化 · MCP 调用 · AGUI 卡片 · 前端渲染。每一步都走协议，不靠 Prompt 调优。

08 · 完整预订

主预订路径：11 个状态，1 条主线

从查询到下单，每一步都有协议支撑、每一步都有兜底。把复杂流程拆解为可分步验证的协议模块。

09 · 技术价值

用协议换复杂度：让流程可维护、可测试、可扩展

状态机约束行为

11 个状态 + OAT 子阶段，Agent 调用有边界，异常可恢复。

Tool Contract 限定

每个 MCP 工具的必填字段、允许阶段全部 schema 化。

Card Schema 强约束

前端按协议渲染，不解析 MCP JSON，不依赖自然语言。

脚本完成确定性工作

日期归一化、阶段守卫、MCP 压缩、卡片组装 —— 全部可测。
