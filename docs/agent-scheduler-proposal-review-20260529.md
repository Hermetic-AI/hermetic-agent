# 《agent-scheduler-proposal》方案评审（新）

> 评审对象：`docs/agent-scheduler-proposal.md`  
> 评审日期：2026-05-29  
> 评审结论：方向可行，但当前版本偏“愿景化”，需补齐可验证技术细节与治理机制后再进入正式立项。

## 1. 总体评价

该方案在“私有化调度中枢 + MCP 工具生态 + 分阶段落地”方向上是正确的，尤其在业务场景贴合（差旅合规、供应商对接、多轮对话）方面有清晰目标。  
当前主要不足是：关键结论缺少可复现证据、部分技术假设过强、核心非功能设计不完整（容量、可靠性、安全边界、观测与运维）。

建议将当前文档从“技术宣讲稿”升级为“可落地设计稿（HLD+NFR）”再推进。

---

## 2. 多维度评审结论

## 2.1 技术选型与论证严谨性

### 优点
- 清晰比较了 OpenCode / Claude Code / Qwen Code / Hermes Agent。
- 选型目标与“私有化、可调度、可扩展”一致。

### 不足
- 存在绝对化表述（如“唯一可纯私有部署”“状态机无法处理某类意图”），缺少边界条件。
- 部分对比项将“框架能力”和“当前实现方式”混为一谈。
- 未给出 PoC 数据支撑（延迟、成功率、并发、稳定性）。

### 优化建议
- 将绝对结论改为“在本项目约束下的最优解”，补充约束清单（合规、预算、团队能力、时效）。
- 增加《选型评分矩阵》（权重 + 打分 + 证据链接）。
- 每个关键结论配最少 1 个可复现实验结果。

---

## 2.2 架构完整性（功能 + 非功能）

### 优点
- 分层清晰：网关、调度中枢、Agent 引擎、业务 MCP、沙箱。
- 已有阶段化交付思路。

### 不足
- 缺少关键运行链路：鉴权模型、租户隔离、幂等、重试策略、熔断/降级。
- Session 与实例绑定关系未定义（会话漂移风险）。
- “MCP Client 同时连多类后端”未定义路由策略（优先级、超时、fallback）。

### 优化建议
- 增加“核心时序图”3套：普通对话、工具调用失败重试、超时中止回收。
- 定义调度策略：`session-affinity`、`least-load`、`latency-aware` 的切换规则。
- 增加故障域设计：引擎故障、MCP 故障、供应商 API 故障的隔离与降级路径。

---

## 2.3 安全与合规

### 优点
- 已重视沙箱隔离与私有化部署。
- 明确了沙箱执行价值。

### 不足
- “数据不出域”与“外部 API 推理”并存，口径冲突。
- 未定义数据分级（PII、财务、行程）及不同级别处理策略。
- 缺少最小权限模型、密钥管理、审计保留周期、合规基线（等保/ISO/SOC）映射。

### 优化建议
- 给出三种合规模式：全私有推理、混合推理、外部推理，并明确适用业务范围。
- 建立 DLP 策略：敏感字段脱敏/令牌化后再进入模型上下文。
- 增加安全控制面：RBAC、密钥轮换、审计追踪ID、工具白名单、命令策略引擎。

---

## 2.4 性能、容量与稳定性

### 优点
- 给出了 P95 延迟与响应目标。
- 规划了多实例池。

### 不足
- 缺少容量模型（QPS、并发会话数、峰值流量、Token 吞吐）。
- 未定义压测方法与 SLO/SLA（可用性、错误率、恢复时间）。
- 流式 SSE 缺少背压、断线重连、消息去重机制。

### 优化建议
- 增加《容量规划表》：`在线用户 -> 并发会话 -> 引擎实例数 -> 成本`。
- 定义 SLO：如 `可用性 99.9%`、`P95 < 8s`、`工具调用成功率 > 98%`。
- 引入统一观测：Tracing（OpenTelemetry）、结构化日志、业务指标面板。

---

## 2.5 业务落地与规则治理

### 优点
- 差旅场景覆盖广，目标明确（查询、预订、合规、报销）。

### 不足
- 合规规则“准确率 >85% / >90%”缺少标注口径与评测集定义。
- 未区分“建议型决策”与“强约束决策”（是否允许绕过）。
- 供应商数据一致性、库存时效、价格波动处理策略未定义。

### 优化建议
- 建立“规则引擎优先，LLM 辅助解释”原则：强约束规则由确定性引擎裁决。
- 定义评测集：按城市等级/员工级别/舱位/酒店标准分层抽样。
- 增加交易幂等与补偿流程：下单重试、重复下单拦截、最终一致性对账。

---

## 2.6 工程可交付性

### 优点
- 三阶段路线图结构清晰。

### 不足
- 每阶段“完成定义（DoD）”不完整，偏功能性，缺少测试与运维标准。
- 缺少版本策略（API 版本、Skill 版本、Prompt 版本、规则版本）。
- 缺少回滚预案与灰度标准。

### 优化建议
- 每阶段补充 DoD：功能、性能、安全、可观测、文档、演练六项达标。
- 建立配置与规则版本化：支持按租户/部门灰度发布与快速回滚。
- 增加演练机制：故障注入、断网、供应商超时、实例崩溃演练。

---

## 2.7 成本模型与ROI

### 优点
- 关注了 token 成本波动问题。

### 不足
- “固定服务器成本”表述过于简化，忽略运维、人力、观测、安全与冗余成本。
- 未做外部 API 推理 vs 本地推理的 TCO 对比。

### 优化建议
- 增加三年期 TCO：基础设施、模型费用、运维人力、安全合规投入。
- 输出成本拐点：何时从 API 推理切换到自建推理更划算。
- 建立成本监控指标：单会话成本、单订单成本、高峰溢出成本。

## 3. 关键风险清单（补充版）

1. 架构风险：会话与实例关系不清导致上下文错乱。  
建议：强制会话亲和 + 会话恢复协议。

2. 业务风险：供应商接口不稳定导致下单失败。  
建议：多供应商路由、熔断、补偿事务、人工兜底。

3. 合规风险：敏感数据进入外部模型。  
建议：数据分级、脱敏网关、高敏场景禁外发。

4. 稳定性风险：SSE 链路长导致客户端体验波动。  
建议：断点续传、事件序号、超时心跳、重放机制。

5. 项目风险：阶段目标过大，跨团队依赖多。  
建议：先做“可交易闭环最小集”（机票查询-合规-下单-回执）。
