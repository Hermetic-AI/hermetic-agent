# 外部日志采集平台接入指南 (OpenAgent Hub 版)

> 模块: OpenAgent Hub (Sanic 25 + Python 3.10+ + structlog 24+)
> 适用: 把 Hub 的运行日志接入公司外部日志采集平台 / SIEM / 检索后端
> 借鉴: `docs/design/外部日志采集平台接入指南.md` (fh-ai 版) — DTO 字段、type 拼接、压缩约定、压缩还原、消费侧伪代码全部沿用
> 真源: `src/openagent/audit/log/*` + `src/openagent/api/http/middleware/log_middleware.py` + `src/openagent/api/lifecycle/lifecycle.py` + `src/openagent/config/settings.py`

---

## 1. 三句话先读懂

1. **三通道并存**:
   - **structlog stdout** — Hub 现有 ~50 处 `logger.info(...)` 继续走 Rich 配色 / JSON line, 给开发期 + docker logs + Loki 直抓
   - **平台日志 (BusiLog/RequestLog)** — 同一个 `logger.info(...)` 通过 `platform_log_processor` 自动转成 BusiLog, 按配置落到 Redis List / 本地文件 / stdout tee
   - **Sanic access log** — `Main |2026-... INFO: 172.18.0.1 GET /health 200` 这种是 Sanic 内置 `sanic.access` logger, 跟 structlog 无关, 关掉: `app.config.ACCESS_LOG = False` (在 `create_app()` 里加)
2. **接入方式 3 条路**: ① `log_use_redis_log=true` → Redis List → **平台订阅消费** (推荐); ② `log_use_redis_log=false` + `log_file_path` → 写本地文件 → **Filebeat / Logkit 采集**; ③ `log_tee_to_stdout=true` (dev) → 平台日志也打到 stdout 跟 structlog 并存
3. **`reqSeqNo` 是唯一追踪主键**: `LogMiddleware` 在每个请求入口生成 (`T<YYMMDDHHMMSS><A-Z><5位序号>`), 放 `ContextVar` (`REQUEST_ID`), 全链路 `logger.info(...)` 自动 merge 进 `BusiLog.reqSeqNo` 字段

## 2. 日志体系架构

```
┌──────────────────────────────────────────────────────────────────┐
│                     OpenAgent Hub (Sanic)                        │
│                                                                  │
│   HTTP 请求 ──► LogMiddleware ──► LogContext.bind_request_id()  │
│                                      (ContextVar: REQUEST_ID)   │
│                                                                  │
│   logger.info("chat_request", session_id=...) ──┐                │
│   busiLogger.info("...", key=val)              ──┤                │
│                                                │                │
│                          ┌──── structlog processor chain ─────┐ │
│                          │ merge_contextvars (注入 REQUEST_ID)│ │
│                          │ add_log_level                     │ │
│                          │ platform_log_processor ──┐         │ │
│                          │ JSONRenderer (stdout)    │         │ │
│                          └──────────────────────────┼─────────┘ │
│                                                     │           │
│   requestLogger.write_start(...) ──┐                │           │
│   requestLogger.write_end(...)    ──┤                │           │
│                                    │                │           │
│   ObjectLogWriter.write_busi()    ◄┴────────────────┘           │
│   ObjectLogWriter.write_request()  ──► 内存 SimpleQueue         │
│   ObjectLogWriter.write_sys()                          (无锁)    │
└────────────────────────────────────┬─────────────────────────────┘
                                     │ RedisWriteTask (asyncio Task)
                                     │   每 0.5s poll 一次
                                     ▼
                           Redis List: "log:openagent"
                           (LLEN 超 maxQueueSize 丢)
                                     │
                                     ▼
                           外部日志采集平台订阅消费
```

## 3. 三类日志对象 (DTO 字段表)

字段名故意保持 camelCase (跟 fh-ai wire format 对齐), 平台消费侧按原始 key 解析, 不要 snake_case 化.

### 3.1 `BusiLog` — 业务日志 (最常用)

来源: `src/openagent/audit/log/dto.py::BusiLog`

| 字段 | 类型 | 来源 | 含义 |
|------|------|------|------|
| `type` | str | `BUSI_LOG_` + `logSystemType` = `BUSI_LOG_OPENAGENT` | 类型 + 系统名拼接 (采集端按此分流) |
| `instanceId` | str | `seq_no.init_instance_id()` (HOSTNAME-hash, 见 §4) | 实例标识 |
| `reqSeqNo` | str | `LogContext.get_request_id()` (LogMiddleware 注入) | **请求流水号 (追踪主键)** |
| `logTime` | datetime | `datetime.now()` | 日志时间 |
| `startTime` | datetime | `LogContext.get_request_time()` 或 `datetime.now()` | 请求开始时间 |
| `level` | str | `INFO` / `WARN` / `ERROR` / `DEBUG` | 日志级别 |
| `programLine` | str | 调用栈最近业务调用点 (`文件:函数(行号)`) | 调用点 (BusiLogger 走调用栈, structlog processor 留空) |
| `threadName` | str | `threading.current_thread().name` | 线程名 |
| `memo` | str | `logInfo` 前 200 字符 | 摘要 |
| `logInfo` | str | 完整内容 (>20KB gzip+base64) | 详情 |
| `infoSize` | int | 原始字节数 | `logInfo` 压缩前大小 |
| `compress` | bool | 是否 gzip+base64 | 详情压缩标记 |
| `logId` | int | `seq_no.get_date_seq_no("LOG_ID")` (20位 int) | 平台 logId (去重主键) |

### 3.2 `RequestLog` — 请求日志 (每请求 2 条: start + end)

来源: `src/openagent/audit/log/dto.py::RequestLog`

| 字段 | 类型 | 来源 | 含义 |
|------|------|------|------|
| `type` | str | `REQUEST_LOG_` + `logSystemType` = `REQUEST_LOG_OPENAGENT` | 类型 + 系统名拼接 |
| `requestTime` | datetime | `datetime.now()` | 请求开始/结束时间 |
| `reqSeqNo` | str | LogMiddleware 生成的流水号 | 流水号 |
| `serviceName` | str | `request.path` | 路由路径 |
| `instanceId` | str | `seq_no.get_instance_id()` | 实例标识 |
| `ip` | str | X-Forwarded-For 优先, 否则 `request.ip` | 客户端 IP |
| `delay` | int | 响应时间 (毫秒), start 时为 -1 | 耗时 |
| `result` | str | `SUCC` / `ERROR` | 结果状态 |
| `errorMessage` | str | 失败时填 `HTTP <status>` | 错误信息 |
| `mainReqSeqNo` | str | 请求头 `main_request_seq_no` | 跨服务调用主链 |
| `bizNo` | str | 请求头 `bizNo` | 业务流水号 |
| `userCode` | str | (预留) | 用户标识 |
| `clientId` | str | (预留) | 客户端标识 |
| `memo` | str | — | 备注 |
| `logId` | int | `get_date_seq_no("LOG_ID")` | 平台 logId |

### 3.3 `SysLog` — 系统/应用日志 (接口已实现, 主路径未启用)

来源: `src/openagent/audit/log/dto.py::SysLog`

`SysLogger` 门面已实现, 但**当前没有主路径调用方**. 启动/停止/异常等系统事件目前走 structlog stdout, 未来需要可接入.

## 4. `type` 字段拼接规则 (grep 锚点)

```python
log.type = base_prefix + logSystemType.upper()
# 例: "BUSI_LOG_" + "openagent" → "BUSI_LOG_OPENAGENT"
```

| 类 | `LogType.value` (基础前缀) | 拼接后 `type` (默认 `openagent`) |
|---|---|---|
| `BusiLog` | `busi_log_` | `BUSI_LOG_OPENAGENT` |
| `RequestLog` | `request_log_` | `REQUEST_LOG_OPENAGENT` |
| `SysLog` | `app_log_` | `APP_LOG_OPENAGENT` |

**环境差异**: 生产环境改 `logSystemType=openagent_prod`, **采集平台索引时用 `type` 前缀 + 中缀分桶**, 不要硬编码 `BUSI_LOG_OPENAGENT`.

---

## 5. `instanceId` 取值规则 (HOSTNAME + 顺序号)

`seq_no.init_instance_id()` 优先级:

1. `init_instance_id(env_override="hub-A")` 显式传 → "hub-A"
2. 环境变量 `AGENT_SCHEDULER_LOG_INSTANCE_ID=hub-A` → "hub-A"
3. **自动**: `socket.gethostname()` 截断 8 字符 + md5(hostname) 前 2 字符 + 当前时间戳 4 位
   - 例 hostname=`openagent-hub-prod-01` → 实例 id = `OPENAGEN-AG5F1432`
4. 兜底: `_INSTANCE_ID` 未 init → `get_instance_id()` 返回 `"UNKNOWN"`

**生产建议**: 容器部署时通过 env 显式给 (例 `AGENT_SCHEDULER_LOG_INSTANCE_ID=hub-A`), 避免 hostname 变化时 instanceId 跟着变.

## 6. 三种接入模式对比

| 模式 | 改动量 | 实时性 | 推荐场景 |
|------|--------|--------|----------|
| **A. 平台订阅 Redis List (推荐)** | 仅配置 `log_use_redis_log=true` | 秒级 (0.5s poll) | 已有 Redis 通道的平台 |
| **B. 文件采集 (Filebeat / Logkit)** | `log_use_redis_log=false` + 配 `log_file_path` | 分钟级 | 走统一文件采集通道 |
| **C. stdout tee (dev only)** | `log_tee_to_stdout=true` | 实时 | 本地开发, 把平台日志也打到 stdout 跟 structlog 并存 |

三种 sink 互斥 + 可叠加策略表 (参见 `object_log_writer.py` docstring):

| `use_redis_log` | `file_path` | `tee_to_stdout` | 实际 sink |
|---|---|---|---|
| true | (任意) | false | Redis |
| true | (任意) | true | Redis + tee |
| false | None | false | drop |
| false | None | true | tee |
| false | 设了 | false | file |
| false | 设了 | true | file + tee |

### 6.1 模式 A: Redis List 通道 (推荐)

**应用侧仅需配置**, 平台订阅 Redis List:

| 配置 | env 变量 (前缀 `AGENT_SCHEDULER_LOG_`) | 含义 |
|------|-------------------|------|
| 开关 | `USE_REDIS_LOG=true` | 启用 Redis 队列 |
| List 名 | `QUEUE_NAME=openagent` | 平台订阅 `log:openagent` |
| 内存上限 | `MAX_QUEUE_SIZE=100000` | Redis 队列满后**丢弃** (跟 fh-ai 一致, 不阻塞主流程) |
| 系统名 | `SYSTEM_TYPE=openagent` | 拼到 `type` 字段后缀 |
| Redis 连接 | `REDIS_HOST/PORT/PASSWORD/DATABASE/TIMEOUT` | 跟业务 Redis 解耦, 独立通道 |
| 轮询间隔 | `REDIS_POLL_INTERVAL=0.5` | 默认 0.5s |
| stdout tee | `TEE_TO_STDOUT=false` | dev 时设 true,跟 Redis 叠加 |

**Redis List 元素格式** (JSON 字符串, 来自 `object_log_writer._serialize`):

```json
{
  "type": "BUSI_LOG_OPENAGENT",
  "logId": 260610182405200002,
  "instanceId": "hub-A",
  "reqSeqNo": "T260610182405B00001",
  "logTime": "2026-06-10 18:24:05.123",
  "startTime": "2026-06-10 18:24:05.100",
  "level": "INFO",
  "programLine": "chat_controller:chat(45)",
  "threadName": "MainThread",
  "memo": "[ToolCall] tool=queryFlightBasic status=SUCCESS",
  "logInfo": null,
  "infoSize": 0,
  "compress": false
}
```

> **压缩处理**: 当 `logInfo > 20KB` 时, `log_compress.maybe_compress` 会 gzip + base64 编码, 置 `compress=true`、`infoSize=<原始字节>`. **平台消费时必须先 base64 解码 + gzip 解压**才能还原详情.

**采集端伪代码** (跟 fh-ai 一致):

```python
# 平台消费端
import asyncio
import base64
import gzip
import json
import redis.asyncio as aioredis


async def consume():
    r = aioredis.Redis(host="...", port=6379, db=0)
    while True:
        _, raw = await r.brpop("log:openagent", timeout=5)
        if not raw:
            continue
        log = json.loads(raw)
        if log.get("compress"):
            log["logInfo"] = gzip.decompress(
                base64.b64decode(log["logInfo"])
            ).decode("utf-8")
        await upload_to_platform(log)


asyncio.run(consume())
```

### 6.2 模式 B: 文件采集

`log_use_redis_log=false` + `log_file_path=/var/log/openagent/platform.log`:

```bash
# Filebeat input
filebeat.inputs:
  - type: filestream
    paths: ["/var/log/openagent/platform.log"]
    parsers:
      - ndjson:
          target: "log"
          overwrite_keys: true
output.logstash:
  hosts: ["logstash:5044"]
```

实现细节 (`object_log_writer.py::_enqueue`):
- 启动期 `open(file_path, "a", buffering=1)` 拿到 line-buffered file handle
- 每次 `write_busi/request/sys` 同步追加一行 JSON (带换行)
- 加 `_file_lock` 保护多线程并发写
- open 失败 (目录不存在 / 权限不够) → 写 warning, 降级到只走内存队列 (不抛)
- `shutdown_log_platform` 关文件句柄
- 跟 Redis 互斥: `use_redis=true` 时不创建文件句柄也不写

### 6.3 模式 C: stdout tee (dev only)

`log_tee_to_stdout=true` (跟 `use_redis_log` / `file_path` 独立):

```
$ docker logs -f openagent-hub 2>&1 | grep platform-log
[platform-log] {"type": "BUSI_LOG_OPENAGENT", "logId": 260610182405200002, ...}
[platform-log] {"type": "REQUEST_LOG_OPENAGENT", "reqSeqNo": "T260610182405B00001", ...}
[platform-log] {"type": "BUSI_LOG_OPENAGENT", "event": "chat_request", ...}
```

- 加 `[platform-log]` 前缀, 跟 structlog 输出区分, 方便 `grep`
- 跟 Redis / file 叠加, 不影响原 sink
- **生产关掉** — 默认 `false`, 开了会双发日志

### 6.4 模式 D: Python logging handler (未实现, 留作扩展点)

外部平台如果提供 Python logging handler, 可在 `logging_setup.py` 加 handler, 转发到 `ObjectLogWriter.write_busi` (走 BusiLog 通道, 而不是 stdout). 当前 Hub 未直接做这层, 但架构上兼容.

## 8. ContextVar 追踪机制 (接入侧不要绕过)

来源: `src/openagent/audit/log/log_context.py`

```python
REQUEST_ID: ContextVar[str]         # 流水号
REQUEST_TIME: ContextVar[datetime]  # 请求开始时间
IS_LOG: ContextVar[bool]            # 是否需要记录 (LogMiddleware 设 True, 业务无需关心)
```

**生命周期**:
- `LogMiddleware.request()` → `bind_request_context(req_id, start_time)` 返回 tokens
- `LogMiddleware.response()` → `clear_request_context(tokens)` 在 `finally` 块调用

**structlog 集成**:
- `structlog.contextvars.merge_contextvars` processor (已在 `logging_setup.py` 的 chain 里) 把这些 ContextVar 合并到 event_dict
- `platform_log_processor` 在 `IS_LOG=True` 时构造 BusiLog, 自动从 ContextVar 拿 `reqSeqNo` 和 `startTime`

**重要约束**:

- **异步任务** (`asyncio.create_task`) 会丢失 ContextVar 上下文, **采集平台追踪会断链**
- 后台任务入口需重新 `bind_request_context(req_id, start_time)` 或 `structlog.contextvars.bind_contextvars(request_id=req_id)`
- 当前 `IS_LOG` 默认 `False`, `LogMiddleware.request()` 主动 `IS_LOG.set(True)`. 如果有路径绕过 middleware (例如启动期心跳), BusiLog 不会落盘 — 这是设计选择, 跟 fh-ai 一致

## 9. 跳过的路径 (中间件白名单)

来源: `src/openagent/audit/log/skip_paths.py`

```python
DEFAULT_SKIP_PATHS = {"/-", "/healthcheck", "/health", "/ready", "/favicon.ico"}
DEFAULT_SKIP_FILE_EXT = {".map", ".ico", ".js", ".css", ".html", ".htm", ...}
```

**不记录请求日志的路径**:

## 10. `result` 字段判定规则 (影响采集端告警)

`LogMiddleware._on_response()`:

```python
status = response.status  # 200 / 4xx / 5xx
result = "ERROR" if status >= 400 else "SUCC"
err_msg = f"HTTP {status}" if status >= 400 else ""
```

| 条件 | `result` |
|------|----------|
| `status_code < 400` | `SUCC`, `errorMessage = ""` |
| `status_code >= 400` | `ERROR`, `errorMessage = "HTTP <status>"` |

> 跟 fh-ai 不同: Hub 端**不**用 `ERROR_CODE` / `ERROR_MSG` response header 判定业务错误 (那是 fh-ai 的内部约定). Hub 直接看 HTTP status. 业务 4xx 错误 (例 scenario 路由失败) 由 `chat_controller` 返回 400, 自动被 LogMiddleware 标 `ERROR`.

## 11. 接入步骤 (推荐模式 A: Redis List)

### 步骤 1: 配 `.env` (或 `docker-compose.yml` env)

**生产 (模式 A + 可选 tee)**:

```bash
# Hub 侧
AGENT_SCHEDULER_LOG_USE_REDIS_LOG=true
AGENT_SCHEDULER_LOG_QUEUE_NAME=openagent
AGENT_SCHEDULER_LOG_MAX_QUEUE_SIZE=100000
AGENT_SCHEDULER_LOG_SYSTEM_TYPE=openagent_prod
AGENT_SCHEDULER_LOG_REDIS_HOST=10.100.30.21
AGENT_SCHEDULER_LOG_REDIS_PORT=6379
AGENT_SCHEDULER_LOG_REDIS_PASSWORD=<secret>
AGENT_SCHEDULER_LOG_REDIS_DATABASE=0
AGENT_SCHEDULER_LOG_INSTANCE_ID=hub-A
# 调试时打开, 生产保持 false
AGENT_SCHEDULER_LOG_TEE_TO_STDOUT=false
```

**本地 dev (模式 B + tee 叠加, 无 Redis 也能看)**:

```bash
AGENT_SCHEDULER_LOG_USE_REDIS_LOG=false
AGENT_SCHEDULER_LOG_FILE_PATH=./work/logs/platform.log
AGENT_SCHEDULER_LOG_TEE_TO_STDOUT=true
AGENT_SCHEDULER_LOG_SYSTEM_TYPE=openagent_dev
```

### 步骤 2: 平台订阅 Redis List

订阅 `log:openagent` (List 名 = `log:` 前缀 + `log_queue_name` 值), 注意:

- **解码 `compress=true` 的详情**: `base64.b64decode` → `gzip.decompress` → UTF-8
- **按 `type` 字段分桶**: `BUSI_LOG_OPENAGENT` / `REQUEST_LOG_OPENAGENT` / `APP_LOG_OPENAGENT`
- **用 `logId` 做去重**: 同一 `logId` 不会重复, 但 Redis 消费至少一次语义需幂等

### 步骤 3: 建索引 (按 §7 表格)

最少必建: `reqSeqNo`, `logId`, `type`, `instanceId`, `level`, `logTime`, `memo`.

### 步骤 4: 错误码归一化 (告警规则)

| `result` + `level` | 模式 | 告警级别 |
|---------------------|------|----------|
| `result=ERROR` 且 status >= 500 | 服务异常 | 紧急 |
| `result=ERROR` 且 4xx | 客户端错误 (含 scenario 路由失败) | 低 |
| `level=ERROR` 但 `result=SUCC` | 业务异常被捕获 (例 `busiLogger.error("task_failed")`) | 中 |

### 步骤 5: 稳定性验收

- [ ] 启动后 0.5s 内 `RedisWriteTask.start()` 触发, 控制台打印 `redis_log_writer_started key=log:openagent`
- [ ] 业务日志可正常落盘 (`busiLogger.info` / `logger.info` → BusiLog)
- [ ] 内存队列 put 异常应丢弃并打印 `log_queue_put_failed`
- [ ] Redis 队列满 (>100000) 应丢弃并打印 `redis_log_queue_full_dropping`
- [ ] 关闭进程时 `shutdown_log_platform()` 优雅停写 + flush 剩余
- [ ] `simplify` (structlog 配置) 切换不影响平台日志 (平台日志走 ObjectLogWriter, 跟 stdout 渲染解耦)
- [ ] 模式 B (file): 启动期 file 句柄创建成功; 写失败只 warning, 不抛; 关闭时 file 句柄关
- [ ] 模式 C (tee): stdout 上有 `[platform-log]` 前缀的 JSON 行, `grep platform-log` 可过滤
- [ ] 模式 A + C 叠加: Redis 收到日志, stdout 同步 tee (生产不开)

## 12. 已知约束 / 反模式

| ❌ 行为 | 后果 |
|---------|------|
| 改 `LogType.value` 枚举值 | 平台分桶规则失效 (字段名要保持 `busi_log_` / `request_log_` / `app_log_`) |
| 改 `seq_no.get_date_seq_no` 的格式 | 流水号格式不一致 |
| 不解压 `compress=true` 直接入库 | 详情全是 base64 |
| 屏蔽 `shutdown_log_platform` | 进程退出时丢日志 |
| 启动前未确认 `log_use_redis_log` 与 Redis 可达 | `LogPlatformSetupError` 快速失败 (跟 Hub 现有 Fatal startup error 模式一致) |
| 采集端把 `memo` 当 `logInfo` 用 | 丢详情 |
| 异步任务不重设 `LogContext` | reqSeqNo 链路断链 |
| 改 DTO 字段名为 snake_case | 平台解析失败 (字段名必须 camelCase) |

## 13. 跟 fh-ai 的差异

| 项 | fh-ai | OpenAgent Hub |
|---|---|---|
| HTTP 框架 | FastAPI | Sanic 25 |
| Redis 客户端 | sync `redis` + Thread | async `redis.asyncio` + asyncio Task |
| `logSystemType` 默认值 | `ai` / `ai_test` | `openagent` / `openagent_prod` |
| `instanceId` 策略 | `app.instanceId` 配置 | `AGENT_SCHEDULER_LOG_INSTANCE_ID` env, 兜底 hostname hash |
| 业务日志入口 | `busiLogger.info(...)` (新 API) | `logger.info(...)` 兼容 + `busiLogger.info(...)` (新 API) |
| 压缩阈值 | 20KB | 20KB (一致) |
| 跳过的路径 | `/-` / `/healthcheck` | `/-` / `/healthcheck` / `/health` / `/ready` / `/favicon.ico` |
| `result` 判定 | status + `ERROR_CODE` header | status only (Hub 不读业务 header) |
| DTO 字段名 | camelCase | camelCase (一致) |
| `type` 拼接 | `BUSI_LOG_` + `ai` | `BUSI_LOG_` + `openagent` |
| Redis 配置 | `log.redis.{host,port,password,database,timeout}` | 同上, 字段顺序一致 |

**DTO 字段和压缩约定 100% 兼容**: 同一平台消费端可以**同时订阅** `log:queueA` (fh-ai) 和 `log:openagent` (Hub), 按 `type` 字段前缀分桶即可, 不需要为 Hub 单独写消费端.
