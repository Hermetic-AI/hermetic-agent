# 差旅 Agent 演示问题库与未来产品方向

## 0. 使用说明

当前已落地的核心能力是 `fh_domestic_flight_booking` Skill，主要覆盖 **国内机票查询与预订流程**：

- 出发地、目的地、日期解析。
- 航班查询。
- 航班结果卡片展示。
- 航班筛选。
- 航班选择。
- 舱位选择。
- 乘机人、出差单、成本中心、联系人等预订链路。
- 政策超规、价格变动、订单预览等流程节点。

因此，演示问题分为两类：

| 类型 | 说明 | 当前适合程度 |
| --- | --- | --- |
| 当前可演示 | 基于机票查询与预订能力，可以直接验证当前 Skill、MCP、AGUI 卡片链路 | 高 |
| 未来规划 | 需要新增酒店、目的地推荐、行程规划、预算、审批、报销等 Skill 后才能完整闭环 | 中 / 低 |

建议演示时明确表达：

- 当前版本重点展示 **OpenCode + MCP + Skill + AGUI** 的技术范式。
- 当前闭环能力聚焦 **国内机票**。
- 多场景复杂问题用于展示未来产品方向：通过新增 Skill 组合能力，而不是改 Agent Core。

## 2. 当前能力可包装出的强演示场景

### 2.1 场景 A：快速查票 + 动态航班卡片

**用户问题：**

```text
帮我查一下北京到上海明天的单程机票，优先给我最便宜、最快和舒适三个推荐。
```

**演示点：**

- 自然语言提取三要素。
- 相对日期归一化。
- 直接调用 `feihe-travel_queryFlightBasic`。
- Hub 自动将 `flightList` 转成 `FLIGHT_RESULT`。
- 前端渲染航班卡片。

### 2.2 场景 B：查询条件不完整 + 表单卡片补问

**用户问题：**

```text
明天帮我查去上海的机票。
```

**演示点：**

- Agent 识别已知字段：目的地=上海，日期=明天。
- 只缺出发城市。
- 使用 `OD_INPUT`，只渲染一个出发城市字段。
- 用户补充后继续查询。

### 2.3 场景 C：从航班卡片进入选择与舱位

**用户问题：**

```text
帮我查北京到广州后天上午的航班，选最快到达的那班，看看有哪些舱位。
```

**演示点：**

- 查询 + fastest plan。
- `chooseFlight`。
- `CABIN_LIST`。
- Tool Contract 保证先选航班再选舱。

### 2.4 场景 D：差旅预订链路

**用户问题：**

```text
给张三订一张下周一北京到深圳的机票，优先选最便宜的经济舱，生成订单预览，不要直接提交。
```

**演示点：**

- 乘机人提取。
- 航班查询。
- 选择最低价。
- 舱位选择。
- 乘机人填写。
- 信息校验。
- 订单预览。
- 明确不直接提交，保持工具边界。

### 2.5 场景 E：政策与价格变动确认

**用户问题：**

```text
帮我订明天上海到北京的航班，如果超差标或者价格变化，先让我确认。
```

**演示点：**

- 政策分支：`POLICY_DECISION`。
- 价格分支：`PRICE_VERIFY`。
- 用户确认后继续。
- Skill 不自行判断政策，MCP/Java 是事实来源。

## 4. 未来 Skill 规划建议

基于当前 `fh_domestic_flight_booking` 的协议化模式，后续可以按 Skill 维度扩展，而不是修改 Agent Core。

### 4.1 推荐新增 Skill

| Skill 名称建议 | 能力范围 | 优先级 |
| --- | --- | --- |
| `fh-domestic-hotel-booking` | 酒店查询、协议酒店、位置筛选、入住离店、订单预览 | 高 |
| `fh-trip-planning` | 多日行程规划、会议时间约束、城市内路线安排 | 高 |
| `fh-travel-policy-compliance` | 差标解释、超规原因、审批建议 | 高 |
| `fh-trip-application-binding` | 出差申请单匹配、成本中心、项目归属 | 高 |
| `fh-itinerary-optimizer` | 多城市路径、时间/价格/舒适度综合优化 | 中 |
| `fh-travel-change-cancel` | 机票改签、退票、酒店取消、损失评估 | 中 |
| `fh-destination-recommendation` | 节假日目的地推荐、预算、热度、天气、偏好 | 中 |
| `fh-ground-transport` | 用车、机场接送、城市内交通 | 中 |
| `fh-expense-summary` | 差旅订单汇总、费用分析、报销前检查 | 中 |

### 4.2 多 Skill 编排目标

未来复杂请求应拆解为多个 Skill 子任务：

```text
用户：端午节给我推荐出行目的地，并规划出行计划，为我订购酒店和机票

Agent Orchestrator
  ↓
fh-destination-recommendation
  输出候选目的地、预算、天气、热度
  ↓
fh-trip-planning
  输出 3 天 2 晚行程
  ↓
fh-domestic-flight-booking
  查询并预订往返机票
  ↓
fh-domestic-hotel-booking
  查询并预订酒店
  ↓
fh-travel-policy-compliance
  校验预算和差标
  ↓
AGUI/AUIP
  输出目的地卡片、行程卡片、航班卡片、酒店卡片、订单确认卡片
```

## 6. 产品方向总结

当前 `fh_domestic_flight_booking` 已经证明了一条可扩展路径：

```text
OpenCode Runtime 不动
  ↓
新增 Skill
  ↓
新增 MCP Tool Contract
  ↓
新增 AGUI/AUIP Card Schema
  ↓
前端按 card_type 渲染
```

未来产品目标可以定义为：

- 从 **单一机票 Skill** 扩展为 **差旅 Skill 族**。
- 从 **查票/订票** 扩展为 **出行规划、酒店、审批、改签、报销前检查**。
- 从 **单卡片航班展示** 扩展为 **多卡片行程工作台**。
- 从 **单轮查询** 扩展为 **跨 Skill 的可恢复任务流**。

最终形态不是把所有能力堆进一个 Agent Prompt，而是将差旅业务拆成一组可插拔、可测试、可维护的协议化 Skill。
