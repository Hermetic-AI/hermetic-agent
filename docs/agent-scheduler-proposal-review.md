# 《基于 OpenCode 智能体引擎的调度中枢技术方案》评审意见

> 评审对象：`docs/agent-scheduler-proposal.md`  
> 评审日期：2026-05-28  
> 评审结论：方案方向正确，但当前版本仍偏“愿景型文档”，落地前需补齐关键技术与工程细节。

---

## 1. 总体结论

原方案在“私有化调度中枢 + MCP 工具化差旅能力”方向上是成立的，分阶段路线也具备可执行框架。  
主要问题在于：核心假设未完全闭环、关键指标缺乏测量口径、架构边界不够清晰，存在“能描述、难验收、难运维”的风险。

建议在立项前补充“架构约束 + SLA/SLO + 安全基线 + 成本模型 + 灰度与回滚机制”五件套，再进入开发阶段。

---

## 2. 多维度评审（问题与优化）

## 2.1 需求与目标维度

### 发现的问题
- 目标混合：同时追求“完全私有化、低成本、高并发、强智能、快速上线”，优先级未排序。
- 业务边界不清：差旅核心路径与非核心能力（多 Agent 协作、自学习、开放平台）在同一阶段并列，范围偏大。
- 验收目标偏结果导向，缺少输入约束，例如“合规准确率 >90%”未定义样本来源与标注规则。

### 优化建议
- 设立明确目标层级：`M0 合规与可用` > `M1 性能与成本` > `M2 智能体验提升`。
- 收敛 MVP 范围：P1/P2 只保留“咨询 + 查询 + 合规拦截 + 下单确认前人工确认”。
- 为每个关键指标补充“定义、计算口径、观测周期、责任人”。

---

## 2.2 技术选型与架构维度

### 发现的问题
- “OpenCode 完全私有化”与“外部 API 推理”在文档中并存，容易引发认知冲突。
- 状态机方案与 Agent 方案对比有价值，但结论略绝对，忽略“混合架构”可行性。
- AgentPool 设计停留在概念层，缺少实例租约、抢占、幂等与重试策略。

### 优化建议
- 拆分两条模式并命名：
  - `Mode A`: 全私有推理（本地模型）；
  - `Mode B`: 私有执行 + 外部推理（过渡期）。
- 采用混合编排：关键交易链路用“可控状态机”，长尾复杂问答用“Agent 自主规划”。
- 补充调度契约：实例心跳、租约 TTL、任务幂等键、重试次数、死信队列策略。

---

## 2.3 安全与合规维度

### 发现的问题
- 安全描述集中在“沙箱隔离”，但缺少主机侧与网络侧的零信任设计。
- 未明确数据分级与脱敏策略，尤其是 PII、订单、财务数据传输路径。
- 审计只提到日志，未定义“谁看、看多久、如何追溯”。

### 优化建议
- 建立安全基线：
  - 主机与容器最小权限；
  - 出网白名单；
  - Secret 托管（KMS/Vault）；
  - 全链路 TLS。
- 定义数据分级（L1-L4）与处理规则（存储、传输、脱敏、保留周期）。
- 设计审计闭环：操作日志、工具调用日志、模型输入输出日志三类分账，满足合规稽核。

---

## 2.4 性能与容量维度

### 发现的问题
- 文档给出了若干延迟目标，但缺少容量模型和压测方法。
- 没有定义并发场景下的退化策略（如高峰限流、降级回答、异步回执）。
- SSE 流式提到“可展示”，但缺失断线重连与背压处理方案。

### 优化建议
- 建立容量公式：`QPS × 平均会话时长 × 工具调用次数` 推导实例数量。
- 设立三层降级：
  - 工具超时降级为“建议型回答”；
  - 非关键工具熔断；
  - 队列化异步处理。
- 补充流式协议细节：`event id`、重放窗口、客户端重连 token、网关超时参数。

---

## 2.5 成本与商业可行性维度

### 发现的问题
- “固定成本优于 token 成本”结论成立前提不足，缺少 TCO 对比模型。
- 外部 API 推理模式下，成本仍高度依赖请求量与上下文长度。
- 缺少环境分层成本（开发/测试/预发/生产）。

### 优化建议
- 给出三种成本曲线：全私有、混合、全云；按 DAU/并发分段比较。
- 增加成本治理手段：上下文裁剪、缓存命中、模板化工具调用、批处理。
- 建立 FinOps 看板：单会话成本、单订单成本、峰值成本预警阈值。

---

## 2.6 可运维性与可观测性维度

### 发现的问题
- 运维项偏宏观，缺少具体 SLO、告警阈值与故障演练机制。
- 未明确跨层 Trace 关联（网关请求 → 调度任务 → 会话 → 工具调用）。
- 对“版本迭代风险”仅提出锁版本，缺少兼容性发布机制。

### 优化建议
- 定义 SLO：
  - 可用性 99.9%；
  - 关键链路 P95/P99；
  - 工具成功率；
  - 合规拦截准确率。
- 建立统一追踪 ID 与 OpenTelemetry 链路。
- 增加发布策略：灰度、金丝雀、回滚阈值、双版本兼容窗口。

---

## 2.7 交付计划与项目治理维度

### 发现的问题
- 阶段三交付项有重复编号与内容重叠，WBS 不够严谨。
- 每阶段里程碑“可验证产物”不足，偏“功能口号”。
- 角色职责不清（架构、后端、算法、测试、运维、合规）。

### 优化建议
- 重排 WBS，消除重复条目，按“必须/应该/可选”分级。
- 每阶段增加 DoD（Definition of Done）：
  - 文档、代码、测试、监控、演练、回滚手册齐备才算完成。
- 补充 RACI 矩阵与周节奏机制（评审、风险清单、变更控制）。

---

## 3. 建议新增的关键章节

建议在原方案后追加以下章节，以便进入实施评审：

1. 非功能需求（NFR）：SLA/SLO、并发目标、可恢复时间 RTO、可恢复点 RPO。  
2. 安全架构：身份、鉴权、密钥、网络、审计、数据分级。  
3. 成本模型：三种部署模式 TCO + 容量预算。  
4. 可观测性方案：日志/指标/链路追踪标准。  
5. 灰度与回滚：版本策略、回滚触发条件、演练计划。  
6. 测试策略：单测、集成、压测、故障注入、红队安全测试。  

---

## 4. 优先级整改清单（建议按两周冲刺执行）

### P0（必须先做）
- 明确私有化边界（是否允许外部推理 API）。
- 给出统一指标口径与验收方法。
- 输出安全基线与数据分级规范。
- 完成 MVP 范围收敛与阶段目标重排。

### P1（应做）
- 完成容量评估与压测方案。
- 补充调度器幂等/重试/死信机制设计。
- 建立 SLO 与告警体系，接入链路追踪。

### P2（可后置）
- 多 Agent 协作高级策略（任务分解器、结果仲裁器）。
- 自学习闭环与模型迭代平台化。
- 开放平台能力与第三方生态接入。

---

## 5. 最终评审意见

该方案具备正确的技术方向和业务价值潜力，但当前文档版本更适合作为“立项草案”，尚不足以直接作为“实施蓝图”。  
在补齐上述 P0/P1 后，可进入小范围试点开发；否则将面临进度失控、验收争议与生产风险外溢。
