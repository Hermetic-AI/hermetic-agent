为什么自建调度层

拥抱成熟开源生态 vs 重复造轮子

自研编排的长期成本

  • AI Agent 技术迭代迅速，自研方案难以跟进最新进展
  • 状态图、ReAct、Tool Call 等范式不断演进，维护成本高
  • 人员更替导致技术债务积累，核心逻辑无人能懂
  • 从零解决 session 管理、并发、超时等工程问题

使用成熟开源 Agent 引擎

  • 专注业务逻辑，调度层只做编排，不碰 Agent 核心
  • 社区推动技术演进，红利自动享受，无需自己跟进
  • SDK 官方维护，API 稳定性有保障
  • 开源项目透明可审计，满足企业合规要求

核心思路：站在成熟开源肩膀上，专注差旅业务深度定制，而非重复造轮子

业务流程定义方式对比

Skill 描述式 vs 状态图代码式

状态图代码开发

  • 每个业务流程需工程师画图、编码、测试
  • 新增航司或酒店品牌 → 修改图结构 + 重新发版
  • 退改签规则变化 → 找到对应节点，改代码，回归测试
  • 业务人员无法参与，只能提需求给工程师
  • 意图分类节点出错 → 后续全错，召回率低

Skill 描述式开发

  • 业务规则以 Markdown 文档定义，接近自然语言
  • 新增航司 → 编写或更新 SKILL.md，Agent 自动理解并执行
  • 差标调整 → 更新 Skill 描述，无需改代码和发版
  • 业务人员可直接参与规则编写和审核
  • Agent 自主理解模糊意图，无需预设分类节点

以"查询北京到上海航班"为例：状态图需要预设意图分类 → 节点A → 节点B；Skill 只需描述"当用户需要订机票时，使用 search_flights 工具"

为什么选择 OpenCode

架构天然匹配调度中枢模式

架构优势

opencode serve 是独立 HTTP 进程，暴露 RESTful 接口。调度层通过 SDK 发起 HTTP 调用，引擎可在另一进程/另一台机器运行，网络透明。

SDK 质量保障

opencode-sdk-python 由官方维护，从 OpenAPI 规范自动生成，API 变更时 SDK 同步更新。会话管理能力完整封装。

SSE 流式响应

client.event.list() 原生支持 SSE 流式事件推送，实时展示 Agent 执行状态，用户体验流畅。

对比结论

Claude Code / Qwen Code 本质是本地进程通信，无法支撑分布式调度。OpenCode 是唯一原生支持远程调度的方案。

核心模块设计

AgentPoolManager

管理多个 opencode serve 实例，维护状态（idle/busy/offline），自动分配空闲实例，支持动态扩缩容

SessionManager

封装 Session API，负责会话完整生命周期：create / chat / messages / abort / revert / summarize / delete

Scheduler

编排任务与实例映射，支持单任务、多任务并行/串行、同 Session 多轮对话、任务超时与中止

MCP Client

连接外部 MCP Server（差旅业务、知识库、沙箱），统一路由工具调用

Skill 层

通过 SKILL.md 定义差旅指令集，Agent 按需加载，Skill/MCP 热加载无需重启

TaskQueue（可选）

基于 Redis 实现分布式任务队列，支持优先级、定时触发、任务重试与失败告警
